### 1. Пустой экран после деплоя (404 для .js/.css)
**Причина**: Nginx (Hestia) перехватывает запросы к статическим файлам и ищет их в `/home/admin/web/DOMAIN/public_html`, а не проксирует к Node.js.

**Симптомы**:
- Белый экран в браузере
- В Network tab: 404 для файлов /assets/*.js, /assets/*.css
- Content-Type: text/html вместо application/javascript

**Решение**: После КАЖДОЙ сборки копировать assets:
```bash
cp -r /var/www/DOMAIN/dist/public/* /home/admin/web/DOMAIN/public_html/
```

**ОБЯЗАТЕЛЬНО** добавить эту команду в скрипт деплоя!

---

### 2. Git ownership error
**Ошибка**: `fatal: detected dubious ownership in repository`

**Причина**: Git не доверяет репозиторию, если владелец папки отличается от текущего пользователя.

**Решение**:
```bash
git config --global --add safe.directory /var/www/DOMAIN
```

### 3. DATABASE_URL must be set / PM2 не видит переменные окружения
**Причина**: PM2 НЕ загружает переменные из .env файла автоматически!

**Симптомы**:
- Ошибка при старте: "DATABASE_URL must be set"
- Приложение падает сразу после запуска
- `pm2 logs` показывает ошибки подключения к БД

**Решение**: Создать `ecosystem.config.cjs` с явными env переменными:
```javascript
module.exports = {
  apps: [{
    name: 'appname',
    script: 'dist/index.cjs',
    cwd: '/var/www/DOMAIN',
    env: {
      DATABASE_URL: 'postgresql://user:pass@localhost:5432/dbname',
      SESSION_SECRET: 'random-secret-string',
      NODE_ENV: 'production',
      PORT: '3001',
      DOMAIN: 'https://DOMAIN'
    }
  }]
};
```

Запуск:
```bash
pm2 start ecosystem.config.cjs
pm2 save
```

### 6. Ошибка авторизации "Unauthorized" (401) после логина
**Причина**: Nginx не передаёт заголовок X-Forwarded-Proto, и Express не знает что запрос пришёл по HTTPS. Secure cookies не устанавливаются.

**Симптомы**:
- Логин проходит успешно, но сразу редирект на /login
- В консоли: 401 на `/api/auth/me`
- В PM2 логах: `Cookie settings: sameSite=none, secure=true`

**Решение**: Добавить в nginx.ssl.conf:
```nginx
location / {
    proxy_pass http://127.0.0.1:3001;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection 'upgrade';
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;   # КРИТИЧНО!
    proxy_cache_bypass $http_upgrade;
}
```

После изменения:
```bash
nginx -t && systemctl reload nginx
pm2 restart appname
```

**ВАЖНО**: Попросить пользователя очистить cookies и войти заново!

// ПОТОМ остальные

### 15. Foreign Key violation после удаления пользователя (КРИТИЧНО!)
**Ошибка**: `Key (user_id)=(X) is not present in table "users"` на таблицах `site_visits`, `page_views`, `express_checks`

**Причина**: Пользователь удалён, но его сессия осталась в таблице `session`. Браузер продолжает слать запросы со старой cookie.

**ВАЖНО - Что НЕ поможет:**
- Очистка логов pm2
- Перезапуск pm2 без пересборки
- Только очистка сессий (нужен и новый код!)

**Решение:**
1. Код должен проверять существование пользователя перед INSERT (функция `getValidUserId`)
2. Пересборка: `npm run build`
3. Очистка сессий: `psql ... -c "TRUNCATE session;"`
4. Перезапуск: `pm2 restart appname`

**См. подробности**: Раздел "Сессия 2026-01-31 (ночь)" ниже

### Сессия 2026-01-31 (ночь) - КРИТИЧЕСКАЯ ОШИБКА: Foreign Key violation при удалении пользователей

**ВАЖНО! Эта ошибка стоила 2+ часа отладки. Прочитай внимательно!**

**Симптомы:**
- Ошибка: `Key (user_id)=(5) is not present in table "users"` 
- Таблицы: `site_visits`, `page_views`, `express_checks` и любые другие с FK на users
- Ошибка появляется ПОСЛЕ успешного удаления пользователя
- Ошибка может появиться через минуты/часы после удаления

**ПРИЧИНА:**
1. Пользователь был удалён из таблицы `users`
2. Но его **сессия осталась в таблице `session`** (connect-pg-simple)
3. Пользователь (или его браузер) продолжает делать запросы со старой cookie
4. Сервер берёт `userId` из сессии и пытается записать в БД
5. FK constraint отклоняет запись - пользователя уже нет!

**НЕПРАВИЛЬНЫЕ подходы (НЕ помогут):**
- ❌ Очистка логов pm2
- ❌ Перезапуск pm2 без пересборки
- ❌ Очистка сессий без пересборки кода
- ❌ Поиск userId в других таблицах

**ПРАВИЛЬНОЕ РЕШЕНИЕ:**

1. **Создать helper функцию `getValidUserId`** в файле где есть доступ к db и users:
```typescript
async function getValidUserId(sessionUserId: number | undefined): Promise<number | null> {
  if (!sessionUserId) return null;
  const [user] = await db.select({ id: users.id }).from(users).where(eq(users.id, sessionUserId)).limit(1);
  return user ? user.id : null;
}
```

2. **Использовать во ВСЕХ местах перед INSERT с userId:**
```typescript
const validUserId = await getValidUserId(req.session.userId);
await db.insert(siteVisits).values({ userId: validUserId, ... });
```

3. **Файлы где это нужно** (все места с insert и userId из session):
   - `server/analytics-routes.ts` - siteVisits, pageViews
   - `server/routes.ts` - expressChecks, audits и др.
   - `server/tools-routes.ts` - toolUsage

4. **При удалении пользователя также удалять его сессии:**
```sql
DELETE FROM session WHERE sess->>'userId' = '5';
```

5. **Команда для деплоя:**
```bash
cd /var/www/securelex.ru && git fetch origin && git reset --hard origin/main && npm run build && psql "postgresql://..." -c "TRUNCATE session;" && pm2 restart securelex
```

**ДИАГНОСТИКА:**
```bash
# Проверить есть ли пользователь в БД
psql ... -c "SELECT id, email FROM users WHERE id=5;"

# Проверить сессии с этим userId
psql ... -c "SELECT * FROM session WHERE sess->>'userId' = '5';"

# Проверить timestamp билда
ls -la /var/www/securelex.ru/dist/index.cjs
```

## ИЗВЕСТНЫЕ ОШИБКИ И РЕШЕНИЯ

### Ошибка 1: Analytics timeline - GROUP BY с dateFormat
**Симптом:** `column "site_visits.started_at" must appear in the GROUP BY clause`
**Причина:** Drizzle ORM неправильно интерполирует JS переменную dateFormat в SQL
**Решение:** Использовать `sql.raw(dateFormat)` с кавычками:
```typescript
// БЫЛО (неправильно):
sql`TO_CHAR(${siteVisits.startedAt}, ${dateFormat})`

// СТАЛО (правильно):
sql`TO_CHAR(${siteVisits.startedAt}, '${sql.raw(dateFormat)}')`
```
**Файл:** `server/analytics-routes.ts`, строки 338-358

### Ошибка 2: Foreign Key при insert в site_visits, page_views, express_checks
**Симптом:** `insert or update on table "site_visits" violates foreign key constraint "site_visits_user_id_users_id_fk"` - `Key (user_id)=(5) is not present in table "users"`
**Причина:** Сессия содержит userId удалённого пользователя, при вставке visit/pageview/check происходит FK violation
**Решение:** Создать helper функцию и использовать во ВСЕХ 3 местах insert:
```typescript
// Helper функция в начале файла
async function getValidUserId(sessionUserId: number | undefined): Promise<number | null> {
  if (!sessionUserId) return null;
  const userExists = await db.select({ id: users.id }).from(users).where(eq(users.id, sessionUserId)).limit(1);
  return userExists.length > 0 ? sessionUserId : null;
}

// Использование во всех insert:
const validUserId = await getValidUserId(req.session.userId);
```
**Файл:** `server/analytics-routes.ts`
**Места:** 
- `db.insert(siteVisits)` - track/visit
- `db.insert(pageViews)` - track/pageview  
- `db.insert(expressChecks)` - track/express-check

### Ошибка 3: Font errors в Playwright/Chromium (НЕ КРИТИЧНО)
**Симптом:** `@font-face src: local() instantiation only available when connected to browser process`
**Причина:** Headless режим Chromium не поддерживает локальные шрифты
**Решение:** Игнорировать - не влияет на работу
